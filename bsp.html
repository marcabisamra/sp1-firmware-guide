06 · The foundation

The sp1-midi BSP

ericlewis/sp1-midi is the minimal public, buildable starting point for SP-1 firmware. It is not a stem player — it's a USB MIDI controller. But it ships working drivers for every peripheral and a clean app skeleton, which is exactly what you want to build on. Prefer to start from a firmware that already plays audio? The tape looper (next) is the other base.

✦ Code of record

github.com/ericlewis/sp1-midi, bundled at assets/sp1-midi-2026-05-13/. Walkthrough: references/19-sp1-midi-bsp.md.

What it does out of the box

Flash it and the SP-1 becomes a MIDI 2.0 controller: faders, buttons, and rockers emit MIDI over USB, with a CDC serial console alongside for debugging.

The architecture you'll extend

The app is a hierarchical state machine built on Zephyr's SMF (State Machine Framework) with a C++ facade. The states are deliberately minimal:

enum AppState {
    STATE_IDLE,          // boot; wait for USB MIDI
    STATE_RUNNING,       // poll faders, route buttons, animate LEDs
    STATE_SHUTTING_DOWN, // mute outputs, prepare for sleep
    STATE_DEEP_SLEEP,    // sys_poweroff()
};

Subsystems are static globals — practical for embedded:

static midi::MidiController g_midi;
static leds::Leds           g_leds;
static power::PowerManager  g_power;

Subsystems you inherit (this is the gold)

SubsystemWhat it gives you
Audio codec driversCS42L42 + TAS2505 brought up on I²C0. Configured at boot — but the BSP never streams audio. You add the I²S feed.
EmmcDriver + StemEmmcDeviceZephyr disk driver over the bit-banged eMMC. Present but unused by the MIDI app — wire it up for playback.
PowerManagerBattery %, charge state, USB attach. Drives the sleep transition.
WatchdogMulti-channel WDT. Register a channel, feed() it, or the device resets — saves you from dead-state bugs.
ResetBreadcrumbsRecords why the last reset happened. Invaluable for debugging crashes without SWD.
Charger driverBQ24232 via GPIO (nPGOOD, nCHG, CE, ISET).
✕ What this BSP deliberately does NOT do
  • No audio playback — codecs are configured but never fed I²S data.
  • No eMMC reads — driver present, app never calls it.
  • No Bluetooth — UART pins defined, no stack.
  • No effects, no auto power-off — long-press is detected but the app must call sys_poweroff() itself.

How to extend it

Add a new state

Per the README workflow:

  1. Add source files to APP_SOURCES in CMakeLists.txt.
  2. Extend AppMachine in AppStateMachine.hpp with your subsystem pointers.
  3. Add new states to the AppState enum and the state table (e.g. AUDIO_PLAYING, EFFECT_ACTIVE).
  4. Initialize your subsystems in main.cpp before app::run().

Add audio playback (the headline extension)

Drop in the reference code

Add audiothingies + storagethingies sources into subsys/ (or a new audio/ tree) and into APP_SOURCES.

Instantiate the engine

Create audio::AudioEngine and storage::DiskManager as globals or AppMachine members; init them after the codecs come up.

Wire the I²S TX path

Use backends/zephyr_i2s_tx.cpp to call AudioEngine::render_block() for each DMA buffer.

Add transport states

An AUDIO_PLAYING state plus a transport sub-state for FF / RW / pause.
