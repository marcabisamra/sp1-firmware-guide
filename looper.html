Case study: the tape looper

In June 2026 someone shipped exactly the thing the rest of this guide builds toward: a complete, buildable, audio-capable custom firmware. chattock/sp1-tape-looper turns the SP-1 into a four-track tape looper — and it's the best public reference for the hard parts (I²S bring-up, real-time flash streaming, a custom on-disk layout). This page reads it as a worked example of the path you've just learned — and, since it's MIT-licensed, as a second base you can fork outright.

▲ This changes the landscape

The rest of this guide was written against a 2026-05-18 snapshot in which the only public buildable firmware was sp1-midi (a MIDI controller, no audio). This firmware postdates that snapshot — its source comments are dated June 2026 (hardware verified 2026-06-05, flash benchmark 2026-06-12). It is, as far as this guide knows, the first public, buildable, audio-capable SP-1 firmware. Treat its specific claims as newer than the skill corpus and verify against the repo. github.com/chattock/sp1-tape-looper

What it does

You feed audio in over USB-C (the SP-1 appears to your computer as a USB sound card), hold the four track buttons to record layered loops, and they play back together through the speaker or headphones. The rocker is tape-style varispeed (speed + pitch together). Loops are stored in the nRF-attached flash and survive power-off and re-flashing. There's a companion web page that moves loops on and off the device as WAV/MP3.

Architecture — three threads, one flash owner

The whole player is one main.c (~2,800 lines) plus an eMMC driver. The design is a clean separation you can copy:

USB-C in ──► [USB ring] ──► audio engine ──► I2S bus ──► speaker / HP
                                 │  ▲
                         record  ▼  │  play
                            [ eMMC flash: one region per track ]
ThreadJobTouches flash?
audio_threadRuns every I²S block (256 frames). Mixes 4 playback tracks + live USB monitor; decimates live input into the one recording track. Never blocks.RAM only
streamer_threadThe only thread that touches flash. Writes the in-progress recording; reads playing tracks ahead of the playhead into per-track SPSC ring buffers.yes — sole owner
midi_thread + mainMIDI-clock housekeeping; an ~8 ms control loop for buttons, faders, LEDs, power, and a USB-serial status line.no
✓ The lesson: single flash owner + read-ahead rings

Real-time audio off a slow, occasionally-stalling flash works because exactly one thread does the blocking I/O and the audio thread only ever reads RAM. The streamer keeps each track's ring filled ~341 ms ahead so the flash's ~150 ms random-access stalls never reach the speaker. This is the pattern your audio page hand-waved as "read ahead to hide housekeeping" — here's it done.

The novel move: USB Audio Class 2 input

Instead of the bootloader's 0x39 album-upload path, this firmware presents itself as a USB Audio Class 2 speaker on Zephyr's next-gen USB stack, with an explicit-feedback regulator driven by USB SOF events. Audio streams in live, in real time — a completely different data path from anything in the protocol chapter.

# prj.conf — the new-stack composite device
CONFIG_USB_DEVICE_STACK_NEXT=y
CONFIG_USBD_AUDIO2_CLASS=y     # UAC2 speaker — real-time audio IN
CONFIG_USBD_CDC_ACM_CLASS=y    # serial console + loop-transfer protocol
CONFIG_UDC_ENABLE_SOF=y        # SOF drives the feedback regulator

USB identity is VID 0x1915 / PID 0x5210 ("softmodded" / "SP-1 Audio") — a third ID pair beyond the factory 2367/1701 and sp1-midi's 1209/0001 from the protocol page. (0x1915 is Nordic's own vendor ID.)

Its own flash layout (not the TE format)

This is a clean illustration of "the eMMC is yours to format." The looper takes the whole card from block 0: block 0 is a metadata header that deliberately overwrites TE's ALBUM_PR index, then 4 songs × 4 tracks of fixed regions.

  • Each track gets a fixed region, rounded up to an 8 KB (16-block) boundary so every flush burst lands on exactly one internal eMMC page — no slow read-modify-write. (This is the same 8 KB-page fact your audio page notes about TE's own sectors, used for a different end.)
  • Tracks are independent — no mixdown is ever stored. An overdub is one region being written while the others are read; mixing happens live and is never written back.
  • Each build stamps a distinct on-flash magic ('SE24' / 'SE48') so switching builds reformats rather than misreading.
◆ "24 kHz" is a storage rate, not the bus rate

Don't misread the build names. The I²S frame is always a true 48 kHz — the CS42L42 codec masters it from the 3.072 MHz oscillator, exactly as the audio chapter describes. The default build records loops mono, decimated by DECIM=2 → 24 kHz to halve flash bandwidth, then interpolates back up on playback. The SP-1 is not "running at 24 kHz."

New hardware info: the sync jack is mapped

✦ Candidate resolution to a known-unknown — verify before trusting

This guide's reference notes flag the second-jack / MIDI GPIOs as not in the published pinout. This firmware drives them, citing "TimK's sync-jack schematic":

  • MIDI on P0.23 (BC807 PNP base → SYNC_RING; the PNP inverts, so a MIDI_INVERT flag compensates). Bit-banged 31250-baud, 24-PPQN clock + Start/Stop, timed off a hardware timer so it doesn't disturb audio.
  • Pocket-Operator sync on P0.20 / P0.17 (PO_A/PO_B → SYNC_TIP), a ~5 ms pulse at 2 PPQN.

If accurate, this resolves a long-standing open question. It either reflects a TimK schematic published after the skill's 2026-05-18 freeze, or the author's own tracing — treat as unconfirmed until checked against TimK directly, and a good entry for questions-for-timk.md.

Robustness — the part worth stealing wholesale

The author treats the "you can brick this and there's no reset pin" reality seriously. The firmware codifies a "BIG FIVE" that maps one-to-one onto this guide's safety rules:

SafeguardHow
App at 0x20000Linked to load above the reserved bootloader region.
Watchdog4 s WDT fed continuously; the final backstop for a hang.
Any fault → clean rebootk_sys_fatal_error_handler turns a bad pointer / stack overflow / panic into sys_reboot(), never a dead hang. MPU stack guard + stack sentinel on.
SYSTEM_OFF → bootloaderPower-off returns control to the bootloader; RESETREAS cleared on boot and before sleep.
Recovery comboHold Track 1 + Track 4 ~1.2 s to drop back to firmware-loading mode — always available.
◆ A telling build note

The author deliberately does not enable -O2 (CONFIG_SPEED_OPTIMIZATIONS): a comment records that the mixer interpolation has latent signed-overflow UB that -O2 "weaponizes" and crashes the device, so it stays on -Os. A good reminder that on this hardware, a green build is not the same as a stable one — test on the device.

One divergence to record, not smooth over

The eMMC access detail differs from the skill's standard line. The skill standardizes on "1-bit mode at 32 MHz" and warns against claiming faster. This firmware's driver header calls itself a 1-bit bit-bang port of TimK's emmc.c, while main.c describes driving it through the nRF's hardware SPI engine at 16 MHz (~1333 blocks/s sustained). Same chip, lower clock, SPI-engine path — not a contradiction, but a genuine second data point worth logging rather than averaging away.

How to extend it

This is the better base for an audio project not just because it makes sound, but because the author left clean seams. The mindset before you touch any of it: this is a complete, real-time system, so the game is extending without breaking, not filling in blanks.

Start with the knobs already exposed

Two compile-time switches in main.c are the highest-value config points — no code surgery needed:

DECIM
2 = 24 kHz (default, more flash headroom), 1 = 48 kHz (full fidelity). It also re-stamps the on-flash magic, so the two rates never misread each other's saves.
SP1_XFER_ENABLE
1 builds in the loop-transfer mode and its web-tool protocol; 0 compiles it out entirely, leaving the proven engine byte-for-byte.

Then pick the seam that fits your change

The three-thread split described above is also the map of where to make a change — each kind of extension has one correct seam, lowest-risk first:

Remap a control

The hold / tap / double-tap gestures and the four faders are read in the ~8 ms main control loop (the function button is P0.27). Re-routing a gesture is RAM-only — it touches neither audio timing nor flash, so it's the safest place to find your feet.

Add a mix-bus effect

The looper ships no effects — only a soft limiter on the summed bus. That makes the per-block mix (256 frames per I²S block, immediately before soft_limit()) the clean insertion point for a filter, drive, or bit-crush — the very DSP the stock player has and this one doesn't yet. It runs every block, so keep it cheap and never let it block.

Change the track / song layout

Track and song counts come from NTRK and the per-slot region map. But the layout is the on-disk format, so any change must also bump the build magic ('SE24'/'SE48') or the firmware will misread old saves. One hard limit: there are exactly four songs because each maps to one of the four status LEDs — more songs needs a new display scheme first.

Extend the transfer protocol

With SP1_XFER_ENABLE on, loops move over the CDC serial link to the companion web page. That protocol — and its WebSerial client — is where you'd add formats, metadata, or a sync feature.

✕ The invariants you can't cross

Every extension above is safe only if you preserve the four properties that make this firmware both work and stay re-flashable:

  • The audio path reads RAM only. Never call flash — or anything that can block — from the audio thread; that's the streamer's job.
  • One flash owner. All eMMC reads and writes stay in streamer_thread; a second writer corrupts loops.
  • The BIG FIVE stay intact. App at 0x20000, watchdog fed, SYSTEM_OFF → bootloader, RESETREAS cleared, and Track 1 + Track 4 recovery always live. There is no reset pin — that combo is your only way back.
  • Stay on -Os. -O2 weaponises a latent signed-overflow bug in the mixer interpolation and crashes the device — fix the UB first or leave the flag alone.

Why it matters for you

  • It's a real, working reference for everything this guide treats as the frontier: codec/I²S bring-up "verbatim from Tim's wiki," the streamer architecture, and a purpose-built flash layout.
  • It proves the roadmap end-to-end — including that an audio-capable build fits comfortably in the 892 KB slot (both .bins are ~83 KB).
  • It updates the landscape: "no public audio firmware exists" is no longer true as of June 2026.
