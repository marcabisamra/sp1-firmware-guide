04 · Protocol

The wire protocol

If you want to flash without the web tool — or build your own uploader — you need to speak the bootloader's serial protocol. It's small: COBS-framed packets, a CRC-8 table, and a handful of opcodes. This is the one area where wrong bytes have burned real AI implementations, so every value here is copied from the solderless source.

✦ Code of record

Packet framing

Transport is USB CDC ACM at 115200 8N1. Every message, both directions, is a COBS-encoded packet terminated by a single 0x00 byte. The decoded packet body is:

+------+-----+-----+------+----------------+------+
| 0x51 | seq | cmd | plen | payload (plen) | crc8 |
+------+-----+-----+------+----------------+------+
   1     1     1     1         plen          1
  • 0x51 ('Q') — fixed start magic on every packet body.
  • seq — host increments per command; device echoes it. Solderless starts at 1.
  • cmd — one-byte opcode (palette below).
  • plen — payload length.
  • crc8 — CRC-8 over [0x51, seq, cmd, plen, payload…] using the table below.
✕ The magic byte is not the opcode

0x51 appears both as the framing magic (start of every body) and as the "end-of-album-upload" opcode. Same byte, different roles. The opcode sits one byte further in, at the cmd position. Don't confuse them.

Command palette (the ones you need)

OpASCIIReplyPurpose
0x52R0x53 SState query. Returns 5-byte state vector, or 4-byte LE32 page counter mid-flash.
0x46F0x47 GFormat / erase app slot. ~5 s. First step of a flash.
0x45EnoneWrite firmware chunk. Payload = [chunk_seq LE32 | flash_addr LE32 | data ≤240B].
0x48H0x49 ICommit / finalize firmware. Called twice at the end.
0x70p0x71 qSet mode (album upload). Payload [1].
0x50PnoneReboot. Re-enumerates USB.
0x377noneReset write counter / start album-upload session.
0x399noneWrite album chunk (136-byte payload, fire-and-forget).
0x43C0x44 DRead write counter (verify chunks received).
0x66f0x67 gVerify album → [isValid, errCode].

There are more (device ID 0x54, battery 0x7A, faders 0x64, LEDs 0x5A…) — the full list is in references/15-bootloader-protocol.md.

The firmware-flash sequence

// from firmware.js::flashFirmware
1. 0x52 (R)              → expect 0x53 (S), any valid reply is fine
2. 0x46 (F)              → expect 0x47 (G); 5s timeout (erases flash)
3. for each 4096-byte page starting at FIRST_PAGE = 0x20:
       for each chunk (≤ FW_CHUNK_SIZE = 240 bytes):
         0x45 (E) [chunk_seq LE32 | flash_addr LE32 | data]   no reply
         sleep 5ms  (100ms at the start of each new page)
4. 0x48 (H) [last_page LE32]  → expect 0x49 (I)
5. 0x48 (H) [counter LE32]    → expect 0x49 (I)
6. device reboots into new firmware

✓ Flash mode is not a separate state

The firmware-flash flow does not require a mode switch first. Send 0x52 once, accept whatever 0x53 reply comes back (4-byte page counter or 5-byte boot-mode "00100"), then go straight to 0x46. The 0x70/0x50 "set-upload-mode + reboot" dance is only for album uploads.

The trap that has actually bitten people

▲ The state-query reply is NOT ASCII

The 0x53 5-byte payload is raw integer bytes (each 0–9), not ASCII characters. Bytes [0,0,1,0,0] mean boot mode; [1,0,5,1,0] mean upload mode. Solderless stringifies with Array.from(payload).join('') — exploiting that (1).toString() === "1" — to get "00100" / "10510".

Using String.fromCharCode(b) produces five control characters instead and the boot-mode check silently fails against real hardware. This exact bug burned the dumper project on 2026-05-14. Either match the reference (Array.from(payload).join('')) or compare the byte array directly.

✕ Other ruled-out claims — do not assert these
  • 0x52 is not a "bootloader version check" — it's a state query, no version anywhere.
  • The 0x39 album packet has no magic / sub-command header. It is exactly chunk_n (LE32) | emmc_byte_offset (LE32) | 128 bytes = 136 bytes.
  • Field two is an absolute byte offset, not a "sector address."
  • The host does not set GPREGRET = 0x1A96 — that was a paraphrase; it's not in any host implementation.
  • 0x39 chunks are fire-and-forget, not ACK'd one by one. A dropped chunk silently corrupts the upload.
