12 · Hands-on

Your first firmware

Putting it together. A concrete, low-risk first project — drive the LEDs from code you wrote — then the staged path from there to a stem player. The goal of the first build is not to be impressive; it's to prove your whole toolchain and flash loop work end to end.

▲ Read this before you flash anything
  • Flashing custom firmware overwrites the stock TE firmware. You can't get it back unless you have a stock binary (not publicly distributed).
  • Develop on a unit you can afford to brick, not your only/favorite one.
  • Your firmware must bring up USB and handle the function-button long-press, or recovery gets harder.
  • Never write below 0x20000. Recovery from a wiped bootloader needs SWD.

The plan

Stand up the toolchain

Install Zephyr + SDK, clone sp1-midi, and build it unmodified first. Confirm you get a build/zephyr/zephyr.bin. Don't flash yet. → Build environment

Prove the flash loop

Enter the bootloader (Track 1+4 + USB-C), flash the stock sp1-midi build via the solderless firmware utility, and confirm the device enumerates as a MIDI controller and as a CDC serial port. Open the serial port and watch for the stem> shell. Now you know your loop works.

Make one visible change you wrote

Edit the LED behavior so it's unmistakably yours — e.g. a startup chase across the 8 LEDs. This is the smallest change that proves your code is running on the device. Code below.

Iterate with logs

Add LOG_INF() calls, rebuild, reflash, read the CDC console. Without SWD this is your debugger. The ResetBreadcrumbs subsystem tells you why the last reset happened — lean on it.

Then, and only then, reach for audio

Once the loop is comfortable, integrate audiothingies / storagethingies and add an AUDIO_PLAYING state. This is a real project, not an afternoon. → Audio engine

Step 3 in detail: an LED chase you wrote

The BSP already initializes the 8 LEDs on PWM2 (Track) and PWM3 (Play) and exposes a Leds helper. The 8 LEDs are: Track 1–4 and Play 1–4. A minimal startup animation in your main(), after the LED setup and before app::run(), might look like this (illustrative — match the BSP's actual Leds API):

// Sweep all 8 LEDs once at boot so you can SEE your code ran.
for (int pass = 0; pass < 2; ++pass) {
    for (int i = 0; i < 8; ++i) {
        g_leds.set(i, 255);     // full brightness
        k_msleep(60);
        g_leds.set(i, 0);
    }
}
LOG_INF("hello from my firmware");   // shows up in the stem> console
✓ Why an LED change is the right first step

It touches no risky subsystem, needs no audio data, and is impossible to misread: either your sweep runs or it doesn't. It also exercises the full pipeline — your edit, the build, the bootloader flash, the running app — which is exactly what you're trying to validate. (It's the same primitive zee_33 used to exfiltrate firmware over the LEDs, so you're in good company.)

Add a new state the BSP way

When you're ready for behavior beyond a boot animation, follow the BSP's extension recipe rather than hacking main():

  1. Add your .cpp to APP_SOURCES in CMakeLists.txt.
  2. Add a state to the AppState enum (e.g. STATE_LED_DEMO) and a row to the SMF state table with entry/run/exit handlers.
  3. Add your subsystem pointer to the AppMachine struct.
  4. Initialize it in main() before app::run(machine).
  5. Register a watchdog channel and feed() it in your run handler so a bug resets the device instead of hanging it.
const int ch = core::watchdog::register_channel("led_demo", 1000);
// ...in your run handler, periodically:
core::watchdog::feed(ch);

Toward a stem player — the honest roadmap

Going from "I can blink LEDs" to "I play my own stems" is the same path the unreleased custom firmwares took. Expect it to take real time — emvee1968 spent about a week with Claude Code to get basic playback.

MilestoneWhat's involved
1 · Read the eMMCWire up the present-but-unused EmmcDriver / DiskManager; read a sector; decode frames with decode_te_frame_payload_i32.
2 · Stream to I²SSet up the I²S TX backend; feed AudioEngine::render_block() per DMA buffer; configure CS42L42 + TAS2505 formats.
3 · Mix + transportPer-stem volume from faders, mute/solo from track buttons, play/stop transport. Add the AUDIO_PLAYING state.
4 · EffectsInstantiate StemEffectRack per stem; wire effect nodes between mixer and codec.
5 · Tape FF/RWThe frontier — block-skipping over the 0/2/1/3 interleave. Nobody has full parity yet; ~2× is realistic.
6 · Custom stemsEncode WAV → album image (per-sector trailer from the audio page) and upload over 0x39.
◆ Using an AI assistant on this? Good — but verify

Two of the existing custom firmwares were built with Claude Code. It's a legitimate path. But this device is a hallucination magnet: an assistant will confidently produce a wrong audio layout, a fake opcode, or an invented workflow step. Cross-check anything it tells you against the cheat-sheet and the don't-hallucinate list, and prefer the bundled source over its memory.
