15 · Accuracy

Don't hallucinate

This page exists because the SP-1 community has publicly corrected AI-generated nonsense about this device — by name, with receipts. If you build firmware with an AI assistant (a legitimate, common approach here), keep this list in front of you. A confidently wrong opcode or byte layout costs hours.

▲ The meta-rule

When you can't verify a factual claim against the bundled source, the public repos, or the skill, the correct answer is "I'm not sure — verify with the live Discord or ask TimK / ericlewis." Hallucinating a specific number, opcode, pin, or behavior is worse than admitting uncertainty.

The three errors TimK called out by name

Word-for-word from Discord #general, 2026-05-10, responding to an AI summary of the Lines archive.

✕ Error 1 — "24-bit PCM little-endian"

Wrong: "The SP-1 stores audio as 24-bit PCM little-endian." TimK: "24 bit PCM little endian … that doesn't make sense."

Right: In memory, right-aligned 24-bit signed in int32_t. On disk, 6 bytes per stem interleaved as L_mid, L_msb, R_msb, L_lsb, R_lsb, R_mid — neighboring bytes belong to different channels. → Audio format

✕ Error 2 — "Python and Rust tools in tkt1000's repo"

Wrong: TimK: "the python and rust tools being in my repo … that doesn't make sense."

Right: github.com/timknapen/SP-1-dev is a documentation repo — README.md, stemplayer_pins.h, images, and a wiki. No tools. The Zephyr BSP is ericlewis's separate sp1-midi. Name the exact repo and file, or don't claim the tool exists.

✕ Error 3 — "Step 4 in the stem-prep workflow"

Wrong: A numbered "prepare stems" procedure. TimK: "step 4 in the workflow to prepare stems is nonsense."

Right: No canonical numbered workflow has been published. Describe the components (stem separation → format conversion → USB upload) and cite who did each. Don't invent steps.

Identity & product traps

TimK ≠ ericlewis
Two different people. TimK = tkt1000 = timknapen (firmware, hardware, bootloader). ericlewis (audio engine, storage, the BSP). Different repos.
SP-1 ≠ PO-32
libpo32 is for the TE PO-32 Tonic (acoustic data transfer). The SP-1 uses USB CDC + eMMC. Don't merge concepts.
Name the firmware
There are 5+ distinct firmwares. "It's in the firmware" is meaningless — say which (stock TE / sp1-midi / emvee1968 / virtualflannel / TimK's).
sp1-midi's app/
It's a MIDI controller. It does not play audio, apply effects, or read the eMMC. Don't look there to explain playback — look in audiothingies.

Protocol-byte traps (these have burned real code)

▲ State-query reply is raw integers, not ASCII

The 0x53 5-byte payload is raw bytes 0–9, stringified with Array.from(p).join('') → "00100" / "10510". Using String.fromCharCode yields control characters and a silent boot-mode-check failure. This bug hit the dumper project on 2026-05-14.

  • 0x52 is a state query, not a "version check." No version exists in the protocol.
  • The 0x39 album packet is exactly chunk_n (LE32) | byte_offset (LE32) | 128B = 136 bytes. No magic, no sub-command, no in-header checksum.
  • Field two is an absolute byte offset, not a "sector address."
  • The host does not set GPREGRET = 0x1A96 — never in any host implementation.
  • 0x39 chunks are fire-and-forget; not ACK'd per packet.
  • Firmware flash does not need a "flash mode" before 0x46.

Hardware / format traps

  • eMMC clock is 32 MHz, 1-bit mode. High Speed is not enabled; don't claim faster.
  • The eMMC is not encrypted — just formatted strangely (8192-byte sectors of 16 blocks).
  • No working public Bluetooth firmware exists. sp1-midi has BT pins, no stack. emvee1968's is unreleased.
  • No per-block trailers in encoder output — one trailer per sector. Tempo = (48000·60)/(24·bpm), not "samples-per-tick."
  • solderless.engineering is online as of 2026-05-18 (it was offline 2026-05-09 to ~05-17). Verify current state if it matters.
  • Don't fabricate Lines post numbers. Cite by author + date if unsure.
✓ A workflow that keeps you honest
  1. Before asserting a byte, pin, opcode, or speed — find it in the bundled source (assets/) or a named-author post.
  2. Cite it inline: [code: path], [Discord #ch, author, date], or [Lines #N].
  3. If you can't find it, say so. Uncertainty beats a confident fabrication every time on this device.
