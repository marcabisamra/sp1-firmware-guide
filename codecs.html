09 · Make sound

Codecs & I²S: making sound come out

The audio page told you what the bytes mean. This one is the step that actually defeats people: bringing up the two codecs and the I²S bus in the right order so a sample written to SDOUT turns into a sound. The trap is the clock topology — get it wrong and you get silence, not an error.

▲ The one thing to internalise

On the SP-1 the nRF52840 is the I²S slave, not the master. An external 3.072 MHz oscillator is the master clock, and the CS42L42 generates LRCLK after you configure it over I²C. So you can't just "turn on I²S and push samples" — the codec has to be alive and its PLL locked first, or there's no frame clock and nothing plays. TKT wiki: I2S, 2026-05-12

The clock chain

The two buses

I²C0 (control)
SDA P1.07, SCL P1.11, up to 400 kHz, 4.7 kΩ pull-ups on both lines
I²S0 (audio)
BCLK P0.12, LRCK P0.11, SDOUT P1.09 — both codecs listen to the same stream
Sample format
48 kHz, 24-bit left-aligned in 32-bit slots, BCLK/LRCLK = 64×, buffer 256 samples (128 frames)
Reset lines
CS42L42 P0.15, TAS2505 P0.09 (NFC1 — needs nfct-pins-as-gpios;). Both active-low.

Both codecs share one I²S stream and each picks the channels it wants via its own register config — the CS42L42 takes a stereo pair for headphones, the TAS2505 takes a mono sum for the speaker. SDIN is unused; the headset mic comes back over I²C status registers, not the I²S wire.

CS42L42 bring-up (headphones)

16-bit register addresses (address byte → reg MSB → reg LSB → data). The order matters:

Clock, then release reset

Oscillator running (P0.13 high), BCLK driven, then release reset (P0.15 high). Wait ≥ 2.5 ms for the internal clock to start before any I²C command.

Confirm it's alive

Read DEVID_AB/CD/E (0x1001–0x1003) — the first sanity check that the chip answers and is the expected part.

Configure and lock the PLL

Write PLL_CTL3 0x1508, PLL_DIV_FRAC2 0x1504, PLL_DIV_INT 0x1505, PLL_CAL_RATIO 0x150A; enable MCLK + ASRC clocks; then poll PLL_LOCK_STATUS 0x130E until locked. No lock → no LRCLK → no audio.

Configure the audio port

Set the ASP (24-bit, 48 kHz — remember nRF is the slave), pick I²S channels with SP_RX_CH_SEL 0x2501, set 24-bit resolution.

Power up the output

PWR_CTL1 0x1101, HP_CTL 0x2001, DAC_CTL 0x240E, then enable jack/mic detect (0x130F, 0x1B75).

✓ The one-register mute

To gate headphone audio without tearing down codec state, write HP_CTL (0x2001): 0x0D = mute all outputs, 0x01 = unmute. Headphone-present check: read TSRS_PLUG_STATUS 0x130F (this is what cs42l42_codec_is_headphone_connected() uses — the tape looper mutes the speaker off this signal).

TAS2505 bring-up (speaker)

Different model entirely: a page + register chip. To write page P register R, first write 0x00 = P (the page-select register, present on every page), then write R = value. Cache the current page to skip redundant selects.

// volume: page 1, register 46 — 0x00 = max, 0x7F = mute
tas2505_write_reg(dev, 1, 46, volume);
  1. Release reset (P0.09 high).
  2. Page 0: configure clocking from BCLK/LRCK.
  3. Page 1: configure the speaker amp path, set volume.
  4. DSP pages: load the default coefficient table (biquad EQ + a likely low-frequency limiter to protect the tiny driver). Entries are { register, b0, b1, b2 } — a 24-bit coefficient per register.
  5. Enable the amp (mute off).
◆ Zephyr does most of this for you

With CONFIG_AUDIO=y + CONFIG_AUDIO_CODEC=y, both chips present the unified audio_codec_configure() / _start_output() / _set_property() API and the drivers handle the register-level dance. After init you mostly just set volume and stream I²S. The catch from the BSP page stands: sp1-midi configures the codecs but never streams — you supply the I²S TX feed.

How the tape looper does it

The case-study firmware follows exactly this clock topology — its source calls the codec/I²S sequence "Tim's wiki sequence, verbatim — native TE topology," and the author's headline design note is the same one at the top of this page: run the board exactly as TE designed it (osc → CS42L42 frame master → nRF + TAS2505 as slaves) and both outputs stay clean. Deviating from that is what produces the classic "it builds, codecs ACK on I²C, but no sound" dead end.

✕ Failure modes to recognise
  • Silence, codecs ACK fine: almost always the clock chain — PLL never locked, or you tried to master I²S from the nRF.
  • I²C NAK right after reset: you didn't wait the 2.5 ms after releasing CS42L42 reset.
  • TAS2505 writes land in the wrong place: stale page cache — you wrote a register without selecting its page.
  • Speaker plays when headphones are in: you didn't wire the 0x130F jack-sense to a speaker mute.
