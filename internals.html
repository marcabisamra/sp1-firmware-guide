Player internals

The two systems that make a real player: getting audio off a slow, stalling flash chip in real time, and the DSP effect rack that runs on it. This is reference-grade detail drawn from the clean-room storagethingies / audiothingies code — what you'd integrate to go past "blink an LED."

The eMMC, and how the nRF talks to it

Chip
Toshiba THGBMNG5D1LBAIL, 4 GB, eMMC 5.0
Bus mode
1-bit (DAT0 only) — multi-bit not used
Clock
32 MHz; High-Speed mode not enabled. Don't claim faster.
Encryption
None — "not encrypted, just formatted strangely"

The driver is a hybrid GPIO + SPIM3 design — bit-bang the command line, then let the SPI engine clock the data:

P0.06 CLK   — GPIO during CMD phase; SPIM3 SCK during data reads
P0.07 DAT0  — GPIO during CMD phase; SPIM3 MISO during data reads
P0.08 CMD   — GPIO bit-bang always (command line + CRC7)
P0.14 VCCQ  — GPIO, enables the eMMC I/O voltage rail
P1.08 RST   — GPIO, active-low reset

Bring-up is an 11-step state machine (Zephyr SMF): POWER_ON → CMD0 → CMD1 poll → CMD2 → CMD3 → CMD9 → CMD7 → CMD8 → CMD16 → VERIFY → SETUP_ASYNC. Every block read is CRC-checked with retry. references/08-emmc-storage.md; assets/storagethingies-2026-05-09/EmmcDriver.cpp

◆ A second data point from the looper

The tape looper drives the same chip through the SPI engine at 16 MHz (~1333 blocks/s sustained), aligning write bursts to the 8 KB internal page. Same chip, lower clock than the skill's 32 MHz figure — log it as a real alternative rather than averaging the two.

Sector layout & the tape trick

The album is stored in 8192-byte logical sectors, each 16 native 512-byte eMMC blocks. A sector holds 340 audio frames (340 × 24 = 8160 bytes) plus a small side-data trailer. The four TE-blocks inside a sector are written in physical order 0 / 2 / 1 / 3:

That ordering is what enables tape-style fast-forward/rewind: at speed, the firmware reads fewer blocks per sector. Measured FF tops out around 2.0–2.5× before you have to start jumping whole eMMC blocks — a frontier no public firmware has fully solved (emvee1968 gave up on it; the looper does varispeed instead). references/11-block-interleaving-tape-fx.md

The streaming model — why audio survives a stalling flash

The flash randomly stalls ~150 ms for its own housekeeping. You hide that with read-ahead: one thread owns the flash and keeps each track's RAM ring filled ahead of the playhead, so the audio thread only ever reads RAM.

This is the architecture both the stock engine and the looper use, and the lesson generalises: exactly one thread does blocking flash I/O; the real-time audio path only touches RAM. The stock firmware's RAM budget is dominated by these prefetch slots (~10 × 8 KB) plus the mixer and delay buffers.

The DSP effect rack

Stock firmware exposes four effect types, each with four variations. Two hard constraints shape the design: only one stem can have an active effect at a time, and activation/deactivation crossfades over one audio block to avoid pops.

✓ The "always-listening delay" trick

Each stem has its own 8192-sample (~170 ms) int16 delay line that is always recording, even when echo is off (a flag tells it to record-but-not-output). So when you switch echo on, it instantly plays back the last 170 ms instead of starting from silence. ericlewis: "because the buffer is always recording, activating the echo effect instantly plays back the past audio history of the stem." Why effects need the embedded timing data

The album carries musical-timing side data in each sector. Per TimK, "the embedded timing data is essential for looping, gate, delay, and 2 of LPF presets" — those effects sync to the beat via a counter the firmware reads, not a free-running clock.

Resolution
24 PPQN — 4 quarters × 24 = 96 steps per bar
MIDI clock byte
0xF8 at 31250 baud, 128 µs per pulse
Where it lives
per-sector trailer at the end of block 0 (the 2026-05-18 encoder writes a clock counter at byte 2040)
✕ Read spec vs write spec — don't conflate

The wiki documents up to 8 bytes of side data per TE-block (2 sync + 2 tempo + 4 LED) — that's what stock firmware can read. The solderless encoder writes a much smaller per-sector trailer (no per-block data). Albums still play. Treat the wiki layout as a read spec, not a thing your encoder must produce — and note the counter-range discrepancy (encoder 49152 vs Galapagoose's older 24,489) is an open known-unknown. references/10-midi-timing-encoding.md

The public audiothingies effect code runs these LFOs/gates free-running; the stock firmware drives them off the sync counter. For stock-feel custom firmware, that's the wiring you'd add — and exactly the kind of detail worth confirming with ericlewis rather than guessing.

What's NOT in the stock rack

No reverb, no pitch-shift, no bit-crusher, no separate compressor/tape-saturation as user effects. The unreleased custom OSes (virtualflannel's beat-repeat / bit-crusher / pitch ±12¢) extend beyond stock — so if you see those attributed to "the SP-1 effects," that's custom code, not the original rack.
