07 · The hard part

Audio engine & format

Making the SP-1 play your stems is the real prize — and the on-disk audio format is where almost everyone (humans and AIs) gets it wrong. This page gives you the exact byte layout, the in-memory representation, and the reference code that decodes both, so you don't have to guess.

▲ The most-cited hallucination on this whole device

"The SP-1 stores audio as 24-bit PCM little-endian." This is wrong, and TimK called it out by name: "24 bit PCM little endian … that doesn't make sense." There are two distinct representations and neither is "little-endian PCM." Get this right and you're ahead of most. Discord #general, tkt1000, 2026-05-10

The 6-byte stem layout (the part that confuses everyone)

Within a single stem's 6 bytes, the left and right channels are interleaved at the byte level. The MSBs of L and R sit next to each other; the LSBs sit next to each other; the mid bytes sit on the outside:

Byte:    0      1      2      3      4      5
Sample:  L_mid  L_msb  R_msb  L_lsb  R_lsb  R_mid
Bits:    8-15   16-23  16-23  0-7    0-7    8-15

This is the authoritative decoder, verbatim:

inline void decode_te_frame_payload_i32(const uint8_t* frame_payload,
                                        DecodedStemFrameI32* frame_out) {
    for (size_t stem = 0; stem < kStemsPerFrame; ++stem) {
        const uint8_t* data = frame_payload + stem * kStemFrameSize;

        frame_out->left[stem]  = (static_cast<int32_t>(static_cast<int8_t>(data[1])) << 16) |
                                 (static_cast<uint32_t>(data[0]) << 8) |
                                  static_cast<uint32_t>(data[3]);

        frame_out->right[stem] = (static_cast<int32_t>(static_cast<int8_t>(data[2])) << 16) |
                                 (static_cast<uint32_t>(data[5]) << 8) |
                                  static_cast<uint32_t>(data[4]);
    }
}

Note the signed-ness trick: the MSB byte is cast to int8_t first (sign-extending into the upper bits) and then shifted left by 16 — giving a 24-bit signed value in the low 24 bits of an int32_t. It is not little-endian, not big-endian; it's a custom interleave where neighboring bytes can belong to different channels.

Three representations — keep them straight

WhereFormat
On disk6 bytes per stem in the interleaved L_mid, L_msb, R_msb, L_lsb, R_lsb, R_mid order above.
In memoryint32_t per channel, right-aligned 24-bit (max ±8,388,607 = 2²³−1).
On the I²S wireDepends on I²S config — typically 32-bit slots carrying 24-bit right-aligned samples.

For float ↔ packed conversion the engine uses float_to_pcm_right24_fast(): clamp to ±1.0, multiply by 8388607.0f. The name says it all — right-aligned, 24-bit. assets/audiothingies-2026-05-09/PcmPacking.hpp

✕ The "correct claim" phrasing

Say: "Audio samples are 24-bit signed. In memory they sit right-aligned in int32_t containers. On disk, each stem stores 6 bytes per frame in an interleaved order — neighboring bytes can belong to different channels." Never say "24-bit little-endian PCM."

The eMMC and the tape trick

The eMMC is driven in 1-bit mode at 32 MHz — High Speed mode is not enabled, and higher clocks are unreliable. Don't try to go faster. The chip is not encrypted, just formatted strangely.

Within each sector the 4 TE blocks are stored in physical order 0 / 2 / 1 / 3. This interleave is what enables tape-style fast-forward/rewind: at speed, the firmware reads less data per sector by skipping blocks. Measured FF tops out around 2.0–2.5× before you have to start jumping whole eMMC blocks — a frontier that no public custom firmware has fully conquered. references/11-block-interleaving-tape-fx.md; tkt1000, Discord #firmware, 2026-05-09

The per-sector trailer (for a custom encoder)

If you write your own WAV→album encoder, here's what the solderless encoders actually write into each sector's trailer (end of block 0). It evolved between versions — the 2026-05-18 form is canonical:

FieldOffset in sectorType
Sync clock (added 2026-05-18)2040uint16 LE, wraps mod 49152
Tempo2042uint16 LE = (48000·60)/(24·bpm)
Per-stem peak envelope ×420444× uint8
✕ Trailer myths to avoid
  • No per-block trailers. The encoder writes one trailer per sector, not four per-block ones.
  • Tempo is the formula above — not "samples-per-tick" and not "Q8.8 BPM."
  • The 2026-05-12 encoder wrote no sync counter at all; only the 2026-05-18 one does, once per sector.
  • The TKT wiki documents a fuller layout (2 sync + 2 tempo + 4 LED per block) that stock firmware can read; encoders ship only the small subset needed for tempo + level meters.
▲ There is no canonical "prepare stems" workflow

An AI once invented a numbered stem-prep procedure; TimK: "step 4 in the workflow to prepare stems is nonsense." Nobody has published a canonical numbered workflow. Describe the components instead: (1) stem separation — emvee1968 used HT Demucs FT; (2) format conversion to the eMMC layout above; (3) USB upload via the bootloader's 0x39 path. Don't fabricate steps between them. references/21-original-firmware-stems.md; hallucination-watchlist.md
