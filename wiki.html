Hardware overview

The nRF52840 is used here not for its well‑known BLE radio — there is no antenna connected to the MCU on the PCB; all Bluetooth is provided by the CYBT module. The nRF has an external crystal for the high‑frequency clock but synthesizes its own LFCLK.

Audio
Everything runs over I²S at 48 kHz, 24‑bit stereo. The master clock is the on‑board 3.07 MHz oscillator marked "A0XRD." TAS2505 drives the speaker (set up via I²C); CS42L42 drives the headphones, generates the I²S LRCLK, and is also set up via I²C; the CYBT module is configured over UART but receives audio over I²S.
General IO
Only one button is read directly by the MCU over GPIO — the Function button. Every other button is part of a resistor ladder read by two ADC pins: (1) Play + the 4 track buttons, (2) Volume up/down + forward/reverse. All 8 LEDs are driven directly by the MCU.
Memory
A 4 GB flash chip stores all audio, connected as 1‑bit eMMC — only DAT0, CMD and CLK are used.
USB
The USB‑C connector is wired for battery charging and the data lines reach the MCU, so it can do CDC ACM (USB serial).
MIDI / PO sync
A second headphone jack can be used for MIDI or Pocket‑Operator sync.

The full, development‑ready pin map lives in src/stemplayer_pins.h. wiki/Hardware-overview.md

Compiling for the bootloader

Rules your firmware must obey to be flashable and recoverable — this is the primary source for the guide's "BIG FIVE":

  • The bootloader writes your application starting at 0x20000; the application's max size is 0xDEFFF.
  • The bootloader starts the watchdog. If your app doesn't feed it, the device resets after 5 seconds.
  • LFCLK, HFCLK and some peripherals — PWM2, PWM3, SAADC — are already started by the bootloader. With the nRF5 SDK, re‑initializing them can make SDK calls fail (it assumes they're uninitialized).
  • The SP‑1 has no hard reset. You must provide a way to enter SYSTEM_OFF to return to the bootloader.
  • RESETREAS must be cleared before going back to the bootloader.
✕ TimK's warning (verbatim intent)

There is no hard reset or reset pin on the SP‑1 and no easy way to power‑cycle it. Fully test new firmware on a device with debugger access before loading it onto a stock SP‑1, and always make sure your firmware can get back to the bootloader. Firmware is uploaded with the solderless.engineering update utility. wiki/Bootloader.md

I²S — audio out

Pins (from stemplayer_pins.h): DOUT P1.09, LRCLK P0.11, SCLK P0.12, OSC_EN P0.13. Configure the nRF I²S peripheral in slave mode — the master/bit clock comes from the 3.072 MHz oscillator and the LRCLK from the CS42L42. Enable the oscillator by driving OSC_EN high, and configure CS42L42 to generate LRCLK.

// key nrf_drv_i2s config for SP-1
config.mode        = NRF_I2S_MODE_SLAVE;
config.mck_setup   = NRF_I2S_MCK_DISABLED;   // external clock
config.ratio       = NRF_I2S_RATIO_64X;      // 64 SCLK / frame -> 48 kHz
config.format      = NRF_I2S_FORMAT_I2S;
config.sample_width= NRF_I2S_SWIDTH_24BIT;
config.channels    = NRF_I2S_CHANNELS_STEREO;

TimK uses buffers of 256 samples (128 frames). wiki/I2S.md

I²C — codec control

TAS2505 is at address 0x18, CS42L42 at 0x48. The PCB has 4.7 kΩ pull‑ups on SDA/SCL and is tested to work with nrfx_twim up to 400 kHz. You must configure a codec over I²C before any audio will play. CS42L42 drives the headphones and provides the I²S LRCLK; its PLL is programmed from registers documented in the Cirrus datasheet (the wiki includes the full power‑up sequence — PLL control, MCLK source select, oscillator‑switch poll until 0x1109 == 0x02).

◆ No mic path

CS42L42 can act as a mic input and its HSIN± pins look wired for it, but there is no audio path back from CS42L42 to the MCU on the SP‑1 (ASP_SDOUT is unconnected). wiki/I2C.md

Bluetooth module — CYBT-353027-02

Connected to the nRF's UART and I²S lines. UART: CTS P1.01, RXD P1.02, RTS P1.03, TXD P1.04; plus SPI_CSN P1.05 and nRST P0.10. TimK notes he hasn't worked on the Bluetooth interface yet — "help is welcome." So there is no public reference for actually driving it. wiki/Bluetooth-Module.md

Storage & the on‑disk audio format

This is the authoritative description the guide's audio page and internals are built on. There is no filesystem — just album metadata followed by a list of songs at known sector offsets.

Sector
8192 bytes (0x2000) = 16 native 512‑byte eMMC blocks.
Sector 0
Reserved for album metadata.
Audio
Starts at the second sector, offset 0x2000.

Sector → block → frame → stem frame

All audio is 24‑bit, 48 kHz, 8‑channel (4 stereo tracks). The nesting:

  • Sector (8192 B) = 340 audio frames (7.083 ms @ 48 kHz) + reserved bytes: 2040–2041 timing/sync, 2042–2043 tempo, 2044–2047 LED data. A sector is 4 × 2048‑byte blocks.
  • Block (2048 B) = 85 audio frames.
  • Audio frame (24 B) = 4 stereo stem frames.
  • Stem frame (6 B) = 2 interleaved L/R samples.
◆ The 0 / 2 / 1 / 3 interleave

Frames are not stored sequentially. They're spread across the sector's 4 blocks in physical order 0, 2, 1, 3 — which is exactly what lets the firmware read fewer blocks per sector at high transport speed (tape FF/RW):

int block_order[] = {0, 2, 1, 3};
for(int frame = 0; frame < 340; frame++){
    int block_id     = block_order[frame % 4];
    int block_offset = 2048 * block_id;
    int frame_offset = 24 * frame / 4;
    int byte_offset  = block_offset + frame_offset;
}

The stem‑frame byte layout (the one people get wrong)

Inside a 6‑byte stem frame the L/R bytes are interleaved unconventionally — neighboring bytes belong to different channels. This is the correct layout (and the reason "24‑bit little‑endian PCM" is a known hallucination — see Don't hallucinate):

OffsetSampleByte
0LMB (mid)
1LMSB
2RMSB
3LLSB
4RLSB
5RMB (mid)
 
// reconstruct right-aligned 24-bit samples in int32
int32_t sample_left  = (data[1] << 24) | (data[0] << 16) | (data[3] << 8);
int32_t sample_right = (data[2] << 24) | (data[5] << 16) | (data[4] << 8);

Album & song metadata

Sector 0 holds the album header, which the tape looper deliberately overwrites when it reformats storage.

OffsetLenAlbum field
013Magic bytes ALBUM_PRESENT
134Album length in 0x2000‑byte sectors
171Number of songs
1864Album title (null‑terminated, padded with 'X' = 0x58)
82*List of song metadata
8192*Second sector — start of audio

A valid album has ALBUM_PRESENT at the very start and as its very last bytes. Each song record: 4 start‑offset (sectors), 4 length (sectors), 64 artist, 64 title. wiki/Album-metadata-format.md
