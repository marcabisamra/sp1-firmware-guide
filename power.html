Power & lifecycle

The most common way to "brick" an SP-1 with custom firmware isn't a bad audio routine — it's forgetting to implement power-off. The device has no hardware reset pin, so a firmware that can't cleanly shut down or get back to the bootloader is a firmware that traps you. This page is the lifecycle you must get right.

▲ ericlewis, in caps, for a reason

"ALWAYS REMEMBER TO PROGRAM IN YOUR POWER DOWN." Discord #firmware, 2026-05-09 emvee1968 shipped an early build with no power-off and had to disconnect the battery every time to get back into the bootloader. Don't be that build.

The power path

USB-C never powers the chip directly. VBUS feeds the BQ24232 LiPo charger, which feeds the system rail, which feeds the 3.3 V and 1.8 V regulators that run the nRF, codecs, eMMC, and BT module.

USB-C VBUS (5V) ──► BQ24232 charger ◄──► LiPo (3.7V nominal)
                         │
                         ▼  system rail
                    3.3V LDO / 1.8V LDO ──► nRF52840, codecs, eMMC, BT, GPIO
✕ Develop with a charged battery connected

The charger expects a battery as its storage element. With the cell disconnected, the regulators see unstable transients on boot and the device may not enumerate. (That instability is also why yanking the battery forces a reset — see recovery below.) Keep a charged cell in the loop while developing.

The BQ24232 — GPIO, not I²C

Unlike its I²C siblings, the BQ24232 is controlled purely by GPIO; charge parameters are set by board resistors, not software. There's no fuel gauge — battery state is inferred from an ADC reading.

SignalPinDirMeaning
nPGOODP0.24inLow = USB power good. High = on battery only.
nCHGP0.22inLow = charging in progress. High = full / fault / no USB.
CEP0.21outDrive low to enable charging (active-low).
ISET overrideP1.00outSwitches charge-current setpoint (fast/slow). Trust the DTS/pin-header P1.00, not the README's P0.19.
Battery senseP0.28 (AIN4)inDivided battery voltage on SAADC ch 4. No fuel gauge — compute % from the voltage curve yourself.

The charger is configured to draw up to 500 mA from USB (set by the ISET resistor network), so the SP-1 charges on any USB port without negotiation. sp1-midi's charger_bq24232.c + PowerManager.cpp wrap all of this. references/05-power-and-battery.md; TKT wiki: Battery-charger, 2026-05-12

The clean power-down sequence

The function button (P0.27, direct GPIO) is the power control; sp1-midi maps a 3-second long-press to INPUT_KEY_POWER. Your handler should tear down in this order, then sleep:

Mute & reset the audio chips

Assert CS42L42 reset (P0.15) and TAS2505 reset (P0.09) so nothing pops or draws current.

Disable Bluetooth

Assert the CYBT reset (P0.10), then de-power its UART.

Stop the LEDs

Stop driving the PWM outputs.

Power down storage

Disable the eMMC supply (P0.14 low) and assert eMMC reset (P1.08 low).

Drop the button rail & the oscillator

Disable the button-common rail (P1.10 low) and the 3.072 MHz audio oscillator (P0.13 low).

Sleep the SoC

Call sys_poweroff() (Zephyr CONFIG_POWEROFF=y) to enter nRF SYSTEM_OFF. The device wakes on USB-attach or a configured GPIO interrupt (function button).

// steps 1–6 are best-effort cleanup; step 7 is the real sleep
codec_assert_resets();        // P0.15, P0.09 low
bt_disable();                 // P0.10 low + UART off
leds_stop();
gpio_low(P0_14); gpio_low(P1_08);   // eMMC supply + reset
gpio_low(P1_10);              // button rail
gpio_low(P0_13);              // 3.072 MHz osc
sys_poweroff();              // SYSTEM_OFF — wakes on USB / button

The looper's "BIG FIVE"

The tape-looper firmware turns the no-reset-pin reality into a checklist it never violates — a good template:

RuleWhy
App linked at 0x20000Never touch the bootloader region.
Feed the watchdog (< 5 s)A hang becomes a reset, not a dead device.
Don't re-init bootloader-owned clocks/peripheralsLeaves the recovery path intact.
SYSTEM_OFF returns to the bootloaderPower-off is always a route back in.
Clear RESETREAS on boot & before sleepClean reset bookkeeping; pairs with reset breadcrumbs.

Recovery when power-off is broken

If you flash a build that can't shut down or reach the bootloader, two escapes:

  • Cold-start the bootloader combo: Track 1 + Track 4 + USB-C from off (the looper also adds a ~1.2 s Track 1+4 in-app recovery).
  • Collapse the rail: with USB plugged in, briefly disconnect the battery — this drops system voltage hard enough to reset the nRF, which then re-enumerates into the bootloader if you're holding Track 1+4. This is the literal escape emvee1968 used.
✓ Build the escape in before you need it

Always wire some always-available path back to the bootloader into your firmware — a long-press power-off and/or an in-app Track 1+4 recovery — before you flash anything experimental. It's the difference between a 5-second recovery and opening the case.
